# 變更紀錄

jt-glogarch 所有重要變更皆記錄於此檔案。

## [1.13.33] - 2026-07-24

### 變更

- **匯入進度列現在會標明每個指標「是誰的」。** 先前只顯示 `Heap`／`Buffer`／`Mem`,看不出來源。現在
  這些指標會分組在 **「目標 Graylog」**（它的磁碟 journal、JVM heap 與 output 緩衝——匯入節流的依據）
  與 **「jt-glogarch 本機」**（這台機器的系統可用記憶體）之下,各自附上滑鼠提示說明。記憶體指標改
  標示為「可用記憶體」以更清楚。

## [1.13.32] - 2026-07-24

### 修正

- **統計報表與系統記錄共用了相同的側邊欄圖示。** 兩者都用了水平線的 `log` 圖示。統計報表現在有自己的
  長條圖圖示（導覽項目、頁面標題,以及報表定義區塊）。

## [1.13.31] - 2026-07-23

### 變更

- **OpenSearch 直連匯出在多索引集時進度不再往回跳。** 匯出現在會先掃描並規劃**所有**索引集（階段
  A）並設定一個固定的總數,再進行匯出（階段 B）。先前分母會隨著逐一到達每個索引集才增加,導致百分比
  進度條可能往回跳;現在改為單調遞增。（取捨:在寫入第一個歸檔前,會先有一段短暫的「掃描中」階段。）
- **跨頁「選取符合篩選的全部」現在只選取 COMPLETED 的歸檔。** 會以 `status=completed` 解析出精確的
  id 集合,讓批次匯入／刪除絕不會作用在損毀／遺失／進行中的資料列上,而且顯示的筆數會反映真正可處理
  的數量。

## [1.13.30] - 2026-07-23

### 修正

於完整應用程式測試與邏輯審查中發現:

- **歸檔／匯入頁面的 CSP 違規。** 匯入對話框的「關閉」按鈕帶有靜態的 `style="display:none;…"`
  屬性,被嚴格的 `style-src 'self'` 內容安全政策擋下（主控台會記錄違規）。已改用 CSS 規則;
  JS 仍透過 CSSOM 切換顯示（符合 CSP）。
- **格式錯誤／被截斷的歸檔標頭會讓讀取器的後備邏輯崩潰,而非優雅降級。** `ArchiveIterator` 的
  中繼資料後備會建立 `ArchiveMetadata()`,但該模型有必填欄位 → 會拋出 `ValidationError`。現在
  改為建立一個優雅的空白佔位中繼資料。
- **強化 indexer failures 自動診斷,避免誤判欄位。** 錯誤訊息解析器原本可能把開頭的
  `[時間戳]`／`[logger]` 方括號誤認為「欄位」而釘選成無用的對應。現在會驗證擷取到的字符確實像個
  合理的欄位名稱（排除時間戳、冒號、空白）,原因解析也能辨識 CamelCase 例外。

## [1.13.29] - 2026-07-23

### 修正

- **每個 OpenSearch 伺服器列現在都有自己的「測試連線」按鈕。** 儀表板的 OpenSearch 卡片會依每個
  Graylog 伺服器列出一列,但先前只有一個共用的測試按鈕（測的是全域設定）,與上方「Graylog 伺服器」
  表格每列都有自己按鈕的做法不一致。現在每一列都有各自的測試按鈕,會測試該伺服器實際使用的
  OpenSearch 叢集（各伺服器自訂設定,或全域後備）。原本每個主機的右鍵測試維持不變。
- **按網址（或其他文字）欄位排序沒有效果。** 前端表格排序的數值判斷太寬鬆:像
  `http://192.0.2.10:9000` 這樣的儲存格會被每一列都解析成數字 `192.0`（parseFloat 在第二個
  小數點就停住）,導致比較結果永遠是 0、列不會重新排序。現在只有當整個儲存格是數字時（可含千分位
  逗號／容量單位）才用數值排序;網址、名稱、認證等欄位一律以文字排序。

## [1.13.28] - 2026-07-23

### 變更

- **匯入時的 indexer failures 現在會自動診斷並修正——不再「請你自己去看 Graylog」。** 先前發生
  合規違規時,只會回報一個數字並要操作人員自行打開 Graylog 的 Indexer Failures 頁面。現在偵測到
  失敗時,jt-glogarch 會讀取失敗明細、解析出**是哪個欄位與哪種錯誤**造成的,並**自動把這些欄位釘選
  為字串（keyword）+ 循環索引**,讓重新匯入時能正確建立索引。作業訊息會指名元凶,例如:*「173 筆
  indexer failures,欄位:'Keywords'（×173）[mapper_parsing_exception]。已自動修復:將 1 個欄位釘選
  為字串並循環索引——請重新匯入以復原受影響的訊息。」*
- **預防性:數值若超出 Java `long` 範圍,會事先釘選為字串。** Windows 事件記錄的 `Keywords` 2^63
  位元遮罩（還原時 indexer failures 最常見的成因）現在會在封存寫入階段就被偵測到,並由 preflight 在
  GELF 送出**之前**釘選為 keyword,讓失敗根本不會發生。（兩項變更皆為非破壞性——keyword 對應不會
  更動已建立索引的資料。）

## [1.13.27] - 2026-07-23

### 新增

- **歸檔頁面的跨頁「選取符合篩選的全部」。** 先前的「全選」只會勾選**當前頁面**,所以按了全選再
  批次匯入,只會匯入那一頁（例如某一天）——很容易誤以為是「所有頁面」。現在勾選一頁後,會出現一個
  明顯的連結——**「選取符合此篩選的全部 N 筆歸檔」**——會用一次請求選取符合目前篩選（伺服器／串流／
  時間範圍）的所有歸檔,跨越所有頁面,並在匯入前顯示筆數確認。設定好時間範圍（輸入或拖拉時間軸）後,
  一鍵就能選取整個範圍。

### 修正

- **原本隱藏的 Shift+全選「所有頁面」模式會忽略目前的篩選**,而選取**整個歸檔庫**而非篩選後的子集。
  現在跨頁選取一律會套用目前的篩選,而且是明顯可見的（不需要 Shift）。

## [1.13.26] - 2026-07-23

### 修正

- **在 Web UI 對話框選的匯入批次筆數／速率被無聲地忽略——匯入一律以設定檔預設值（批次 500）
  執行。** 在匯入對話框設定例如「每批 50 筆／100 毫秒」,實際匯入卻用 500。根本原因:
  `Importer.import_archives` 會無條件地把流量控制物件的 `batch_size` 與 `rate_ms` 覆寫成設定檔
  預設值,即使 Web UI 在傳入前已經把使用者選的值設定好了。現在只有當呼叫端（CLI／排程）未提供
  流量控制時才套用設定檔預設值;Web UI 對話框選的批次筆數與速率會被採用。匯入進行中的「批次」
  選擇器（會反映實際執行中的值）現在會正確顯示所選的筆數,而不是 500。（CLI／排程匯入行為不變,
  仍從設定檔取得批次／速率。）

## [1.13.25] - 2026-07-23

### 修正

- **OpenSearch 直連匯出在匯出超過一個索引集時，進度顯示的「已處理／總數」不合理且卡在
  99％。** 匯出會逐一走過每個索引集前置碼，先前會把作業的分母（`messages_total`）重設為
  「目前這個」前置碼的文件數，但已處理數是跨所有前置碼累計的。因此多索引集匯出時會顯示成
  例如 `214,981,759 of 21,160,634 records`（已處理遠大於總數），進度條卡在 99％。現在分母
  改為**累計所有前置碼的總文件數**，讓整個作業維持「已處理 ≤ 總數」，百分比也是真正的整體
  進度。**這是純顯示問題——沒有任何資料遺失或漏匯，每個索引集的每個索引一直都有被匯出。**
  （單一索引集匯出與 API 模式匯出不受影響。）

## [1.13.24] - 2026-07-23

### 變更

- **systemd 記憶體上限改為僅「軟性」（`MemoryHigh`），不再硬性強制終止。** 先前出貨的
  unit／升級 drop-in 還會設硬性 `MemoryMax=4G`，這會讓核心對 jt-glogarch 的整個 cgroup
  進行 OOM 強制終止——而 PDF 報表用的 Chromium 就在同一個 cgroup，繁重的報表算圖可能因此
  被殺掉。現在只設 `MemoryHigh=3G`：超過時核心會限速 jt-glogarch 並回收其記憶體（保護同機的
  OpenSearch／Graylog 與整台機器），但不會終止行程。`upgrade.sh` 會自動將既有的硬性 drop-in
  遷移為軟性。此為額外的保險絲——真正的防護是**串流封存讀取器**（匯入記憶體與封存大小無關而
  保持平穩：實測 50 萬筆封存約 47 MB，舊的整包載入則約 545 MB）加上**匯入記憶體守門員**，
  兩者皆已出貨。

## [1.13.23] - 2026-07-23

### 新增

- **「疏通目標 Graylog」——一鍵、非破壞性地解除卡住的目標。** 當大量匯入把目標 Graylog
  塞住時（journal 堆積、環形緩衝卡在接近上限、或寫入索引無法再接受寫入），操作人員現在可以
  在**不動到任何訊息資料**的前提下疏通它。它會**循環寫入索引**（循環 deflector，讓新的索引
  接手寫入）並**重建索引範圍**（修正「資料明明在、但搜尋找不到」），並顯示 journal 與緩衝的
  前後對照,讓你看到它正在消化。此操作**不會刪除任何訊息、索引或索引集**——確認對話框會明確
  說明這一點。提供兩個入口：**匯入進度畫面**（於匯入進行中疏通該次匯入的目標）與**設定頁**
  每一台伺服器旁（日常維運）。新增端點 `POST /api/graylog/flush`；同時支援已設定的伺服器與
  臨時的匯入目標（匯入目標的密鑰會與已儲存的預設值調和）。
- **發布關卡新增 headless 瀏覽器 UI 冒煙測試**（`scripts/ui-smoke.py`）：以真實的 headless
  Chromium 載入 UI、驗證 i18n 已就緒、切換為 zh-TW,並在任何未捕捉的頁面／主控台錯誤時
  **直接讓發布失敗**。`node --check` 只能抓 JavaScript 的**語法**錯誤,此測試還能抓到執行期／
  渲染的問題。

## [1.13.22] - 2026-07-23

### 修正

- **嚴重：整個介面退回英文、設定頁一片空白（自 1.13.20 起的回歸）。** 某個 i18n 單引號字串裡
  混進了撇號（`… the target's buffers …`），造成 JavaScript **語法錯誤**，`i18n.js` 無法解析
  → 翻譯從未套用（頁面只剩英文預設值）、靠 JS 渲染的設定頁也炸掉。**沒有任何資料遺失**——這不會
  動到 `config.yaml`；把字串修好，語言與所有設定就全部回來。
- **新增發布關卡：靜態 JavaScript 現在會做語法檢查（`node --check`），只要有壞掉的 `.js` 就
  直接讓發布失敗。** 先前 Python 測試從不載入瀏覽器 JS，才讓它漏出去；以後不會再發生。

## [1.13.21] - 2026-07-23

### 修正

- **匯入不再把整台記憶體吃爆——歸檔讀取改為真正的串流。** `ArchiveIterator` 宣稱是串流，實際卻
  對整份歸檔做 `json.load()`：50MB 的 `.json.gz` 轉成 Python 物件會膨脹到好幾 GB，匯入大歸檔時
  jt-glogarch 記憶體會漲到 ~12GB，被核心 OOM killer 殺掉（**「Interrupted by service restart」、
  停在可重現的筆數**），同機部署還會連 OpenSearch 一起害到。現在改用 `raw_decode` 一次讀一筆，
  記憶體只留約一個批次（＋256KiB 視窗）。**這是匯入 OOM 當機的根因修正。**
- **暫停/塞住的匯入現在按取消有效。** journal 自動暫停原本是不可中斷的 30 秒 sleep，退讓期間按取消
  沒反應；現在每秒檢查取消旗標。GELF 送出也加了 connect（15 秒）與 drain（30 秒）逾時——目標 TCP
  緩衝滿（Graylog 塞住）時原本會無限卡在送出、取消永遠進不去；兩種情況現在都能即時取消。
- **匯入進度記憶體洩漏。** 匯入進度事件清單無限成長（匯出早有修剪），百萬級匯入會撐大記憶體；現在
  只留最後約 100 筆。

### 新增

- **匯入的本機記憶體守門。** jt-glogarch 通常與目標 Graylog＋OpenSearch 裝在同一台 VM，大匯入會吃光
  共用記憶體觸發 OOM。匯入現在會讀 `MemAvailable`，可用記憶體過低就暫停（`import.mem_pause_mb`
  預設 700MB；`mem_slow_mb` 預設 1400MB 放慢）。可用記憶體會即時顯示在 journal/buffer/heap 標籤旁。
- **批次間隔改成可用鍵盤輸入的數值欄位**（對話框與進度畫面），不再只有滑桿；滑桿保留、雙向同步。
- **systemd 記憶體上限（縱深防禦）。** 升級／安裝現在會加一個 drop-in（`MemoryHigh=3G`、
  `MemoryMax=4G`），就算 jt-glogarch 真的失控暴衝，也只會在它「自己的 cgroup」內被 OOM 處理，
  **絕不會拖垮同機的 OpenSearch／Graylog 或整台 VM**。大機器或大量產報表可自行調高。

## [1.13.20] - 2026-07-23

### 修正

- **匯入絕不可以把目標 Graylog 卡住。** 節流現在會盯著 **process／output ring buffer**——這是
  Graylog 寫不進 OpenSearch 的最早訊號（buffer 約 65K 就塞滿，遠早於 journal 出現大量堆積）。
  buffer ≥ 70% 放慢、≥ 90% 暫停，從源頭避免匯入把 journal 灌爆、把 Graylog 卡死。
- **Preflight 會拒絕把資料匯入「已經塞住」的 Graylog。** 若 process／output buffer 已 ≥ 90%
  （OpenSearch 跟不上），匯入會被擋下並明確提示（≥ 70% 則警告）——不會讓你啟動一個注定會卡住、
  還把 Graylog 弄更糟的匯入。
- **JVM heap 節流不再對「健康的 Graylog」誤判。** Graylog 用 G1GC 本來就常態在 80–95% heap，
  舊的 80／92% 門檻會害「每一次」匯入都被節流。heap 現在只當高位近 OOM 的保險（≥ 95% 放慢、
  ≥ 98% 暫停），真正可靠的訊號是 ring buffer。
- **匯入進度記憶體洩漏。** 匯入的進度事件清單會無限成長（匯出早就有修剪）；百萬級匯入可能把服務
  記憶體撐爆。現在跟匯出一樣只保留最後約 100 筆。

匯入畫面上：journal 標籤旁現在也會顯示 **Buffer%** 與 **Heap%**，一眼看出是哪個訊號在節流。

## [1.13.19] - 2026-07-23

### 修正

- **匯入的批次筆數下拉（進度畫面那個）無法調整。** 新加的下拉被 App 的自訂下拉元件包住了；需要
  互動的 `<select>` 都要加 `class="no-custom"`（其他 `data-act-change` 的下拉都有）。現在對話框
  與進度畫面的批次下拉都改回原生、可正常調整。
- **進度畫面的「批次筆數」標籤在窄欄位裡換成兩行**——進度畫面改用短標籤「批次」並設不換行，且下拉
  會反映「實際執行中的批次筆數」，不再顯示過時的值。

## [1.13.18] - 2026-07-23

### 修正

- **目標 Graylog journal「卡住」時，匯入現在會先暫停，而不是只在滿出來時才反應。** 先前節流只看
  絕對堆積量（≥ 10 萬放慢、≥ 50 萬暫停、≥ 100 萬停止）。現在：
  - **卡住（沒在消化）**：堆積量偏高且相較上次取樣「沒有下降」，會從「放慢」升級為「暫停」——不再
    對一個 Graylog 無法寫進 OpenSearch 的 journal 一直灌。
  - **匯入途中 journal 讀不到**：若原本讀得到、之後突然失敗（目標不可達／出錯），匯入會暫停（fail-safe），
    等恢復再自動續傳，而不是盲目繼續灌。若目標從頭就沒有 journal 端點則豁免（不會卡死；以使用者速率
    執行並警告一次）。
  - **檢查改成以「訊息數」為節奏（約每 5000 筆）**，不再固定每 10 批，避免大批次留下過大的盲區。

## [1.13.17] - 2026-07-23

### 新增

- **GELF 匯入：可選批次筆數（50／500／1000／2000），匯入對話框與匯入中進度畫面「兩邊都能調」。**
  批次太小匯入很慢；現在可先選起始批次筆數，匯入進行中也能即時調整（後端 `/import/{id}/rate`
  本來就吃 `batch_size`，只是 UI 沒開出來）。預設維持 500。
- **匯入節流現在也會判斷目標 Graylog 的 JVM heap**，不再只看 journal 堆積。journal 監控會取樣
  `/api/system/jvm`，取「journal 與 heap 兩個訊號中較嚴重者」來退讓（heap ≥ 80% → 放慢、≥ 92%
  → 暫停），避免大批次把目標端壓到 OOM。heap % 會即時顯示在 journal 標籤旁。

## [1.13.16] - 2026-07-22

### 新增

- **匯出作業顯示 index set 涵蓋範圍標籤。** OpenSearch 匯出作業現在會在「作業歷程」與匯出結果對話框
  顯示彩色標籤：全部涵蓋時為綠色 **「涵蓋所有 index set」**，被限縮而略過部分時為琥珀色

## [1.12.0] - 2026-07-07

### 新增——選用的歸檔防竄改（HMAC，需自行啟用）

新增一層**選用**的完整性機制（預設關閉），能偵測「蓄意竄改」而非僅「損毀」。裸 SHA256
只能偵測相對於已存值的變動——能同時改「歸檔檔案 + DB 裡的 checksum」的人就能偽造一組
對得起來的假值。此功能加上**帶秘鑰的 HMAC-SHA256**：沒有秘鑰就無法對竄改後的內容算出
正確 MAC，因此就算同時改檔案與 DB 也過不了驗證。

- **金鑰來源**(可插拔)：環境變數 `JT_HMAC_KEY`(base64／hex)優先，否則用
  `integrity.hmac_key_file`(預設 `/opt/jt-glogarch/.hmac_key`，權限 0600)。若要連 root／
  服務帳號都防，請勿把金鑰檔留在機器上——僅在封存／驗證時以環境變數提供，並把 ledger
  存到機器外。
- **獨立雜湊清單(ledger)**：每個封存歸檔的 `(路徑, sha256, hmac, 大小, 封存時間)` 會寫入
  `integrity_ledger` 表；`glogarch integrity-manifest` 可匯出到機器外保存，即使 root 把
  檔案與 DB 全改了，仍能拿機外副本對帳揪出。
- **驗證會偵測竄改**：`glogarch verify`(與排程驗證)會回報獨立的 **`TAMPERED`** 狀態
  (HMAC 不符)，與 `CORRUPTED`(SHA256)區分，UI 顯示紅色徽章並發送通知。
- **新增 CLI**:`integrity-init`(產生金鑰)、`integrity-seal`(封存既有歸檔——僅能證明
  「從現在起」未被動，無法回溯證明過去)、`integrity-manifest`(匯出 ledger)。
- 關閉時**完全 no-op**——維持現有行為(僅 SHA256)，不受影響。

## [1.10.7] - 2026-07-05

### 匯出安全

- **自適應反壓守護（API 與 OpenSearch 直連皆適用）。** 大量匯出會對 Graylog 用來索引的同一個 OpenSearch 叢集造成負載；在繁忙或 HDD 儲存上會餓死 ingestion，甚至把 Graylog 卡到需重啟。匯出現在以**固定約 15 秒節奏**（下沉到每個 batch，與 chunk 大小脫鉤）讀取 Graylog 自身的健康訊號——JVM heap%、disk journal（未提交筆數）、input／process／output 環形緩衝區——並**在任一訊號持續上升時暫停，等降下來才續跑**。JVM heap 採**兩段式**：軟門檻（75%、須持續）遠早於天花板就退載又不會被 GC 尖峰誤觸，硬上限（90%）遇突發飆高單次即暫停。且為 **fail-safe**：讀不到 Graylog（正是它出狀況的當下）時暫停而非繼續猛打。每次暫停都寫入記錄，執行中的作業會顯示是哪個訊號高負載。連線持續失敗則由斷路器中止。所有門檻可在 `config.yaml` 的 `export:` 下調整。
  - *修正舊守護的盲點：* 舊版只看 JVM heap 且門檻高達 85%（快撞天花板）、每 5 個 chunk 才檢查（相隔數分鐘）、且在 Graylog 連不上時回傳「0%（健康）」，導致對著已凍住的伺服器繼續匯出。
- **連線測試顯示 Graylog heap 建議。** 測試 Graylog 伺服器時會回報目前 JVM heap（`-Xmx`）與建議最小值，讓 heap 配置不足在造成匯出停擺前就被發現。

## [1.9.1] - 2026-07-04

### 修正 — 匯出資料完整性總體檢（Graylog API 與 OpenSearch Direct）

針對從 1.7.9 升級的客戶，對兩條匯出路徑進行完整稽核。下列問題在高負載時皆可能無聲地遺失或重複訊息，而作業卻仍回報綠色「已完成」。

- **Graylog API — 同一毫秒邊界資料遺失（高）。** 當時間視窗填滿 10,000 筆結果上限時，下一個視窗以「最後時間戳 + 1 毫秒」推進，無聲地丟棄所有與該毫秒相同、但超出上限的訊息。現在改為推進到「最後一筆訊息的精確時間戳」，並以 `gl2_message_id` 對邊界毫秒去除重複，確保不遺失也不重複。此修正需要以**毫秒精度**送出搜尋視窗（原本被截斷到整秒，也會導致邊界那一秒被重複抓取）。
- **Graylog API — 視窗提早結束（高）。** 分頁原本以每次請求回傳的 `total_results`（可變值）判斷結束，若索引在匯出過程中輪替或因保留策略刪除，該值會縮小而丟失視窗尾端資料。現在改以「短頁」（可靠的資料結束訊號）判斷結束。
- **Graylog API — 異常時間戳導致無聲截斷（中）。** 無法解析或停滯的邊界時間戳原本會 `break` 並無聲截斷該區塊。現在改為丟出錯誤（會被呈現並於下次重試），且時間戳解析器可處理任意 ISO-8601 格式（奈秒、明確時區位移）。
- **OpenSearch Direct — 暫時性 5xx 未故障移轉（高）。** 某節點在重試耗盡後回傳 `429/500/502/503` 時，原本會中止整個索引；單一過載節點（斷路器 / GC 暫停）即使叢集中另有健康節點，也會無聲丟棄整個索引。現在會故障移轉至下一台主機，只有在**所有**主機都耗盡時才丟出錯誤。
- **OpenSearch Direct — `search_after` 提早停止（中）。** 分頁原本以預先計算的 `total` 判斷結束，該值可能因並行的段合併 / 重新整理而低估，導致跳過尾端。現在改為分頁到游標耗盡為止。
- **兩種模式 — 部分失敗不再隱藏於綠色「已完成」之後。** 每個索引 / 區塊的失敗現在會記錄於作業備註（`⚠ N 個失敗 — 下次執行將重試`）。執行中途的系統性失敗（某次成功後叢集轉為 RED）現在會快速中止，而非空轉數小時。OpenSearch 匯出期間磁碟耗盡現在視為致命中止，不再被當成單一索引錯誤而在每個剩餘索引上重試。
- **速率限制器**不再於等待期間持有鎖，原本此行為會將所有請求序列化並使突發配額失效。

### 修正 — 升級 / 移轉（1.7.9 → 1.9.1）

- **啟動時復原卡在 `IMPORTING` 的封存。** 匯入若於執行中被中止（例如升級時的服務重啟），會使封存列卡在 `IMPORTING`，而每封存鎖會使其永遠無法再匯入。啟動時現在會將停滯的 `IMPORTING` 列翻回 `COMPLETED`，落實文件所述的當機復原保證。

### 修正

## [1.7.14] - 2026-05-14

### 修正 — Ubuntu 24.04 全新安裝執行 `deploy/install.sh` 時失敗於 `externally-managed-environment`

- 客戶於 Ubuntu 24.04.4 LTS（Python 3.12）的全新環境跑 `sudo bash deploy/install.sh`，在「Installing jt-glogarch and dependencies...」階段就中止，錯誤訊息為 `error: externally-managed-environment` / 「This environment is externally managed」，並建議改用 `apt install python3-xyz`、venv 或 `pipx`。install.sh 還沒走到後面的目錄建立、SSL 憑證、systemd 服務等步驟，系統被留在半成品狀態。
- 成因：Ubuntu 24.04（與 Debian 12+）依 PEP 668 規範隨機附帶 `/usr/lib/python3.12/EXTERNALLY-MANAGED` 標記檔。當這個檔案存在時，`pip install <pkg>` 對系統 Python 預設拒絕安裝，必須額外帶 `--break-system-packages` 旗標才會放行。`deploy/install.sh`、`deploy/upgrade.sh`、`deploy/uninstall.sh` 是在 PEP 668 還沒普及前寫的，從未帶過這個旗標。Ubuntu 22.04（Python 3.10/3.11）沒有這個標記檔，所以同樣的腳本在 22.04 一直能跑 — 問題只在 24.04 客戶第一次裝時才浮現。
- jt-glogarch 是專屬單一用途的服務型安裝：以自己的 system user 執行、systemd unit 呼叫 `/usr/local/bin/glogarch`、預期該主機**不會**並行跑其他 Python 應用。把套件寫進系統 Python 本來就是設計上預期的部署模式，因此 `--break-system-packages` 是正確的旗標。（要改用 venv 會牽動 systemd unit 路徑、所有文件參照、upgrade 流程，影響面大，**不在此次修法範圍**。）
- 修法：在每個 deploy 腳本開頭偵測 EXTERNALLY-MANAGED 標記檔；存在就把 `--break-system-packages` 自動加到所有的 `pip install` / `pip uninstall` 呼叫上。舊版發行版沒有標記檔的話行為完全不變。偵測程式碼跨發行版可移植：
  ```bash
  EM_FILE=$(python3 -c 'import sysconfig; print(sysconfig.get_paths()["stdlib"] + "/EXTERNALLY-MANAGED")' 2>/dev/null || true)
  PIP_FLAGS=""
  ```

### 修正 — 長時間 OpenSearch 匯出隨機出現「cannot start a transaction within a transaction」，整個索引被跳過

  - 新增 `ArchiveDB.backfill_audit_usernames(ip_user_pairs, default_user)` — 取代 `audit/listener.py::_refresh_ip_user_cache` 中逐 IP 的迴圈 UPDATE 與最後 commit。

  - 新增 `tests/test_concurrent_db_writes.py`，含四個迴歸測試，分別讓多執行緒對新封裝與 `update_job()` 密集競爭 1–2 秒。舊的繞鎖寫法在這些測試下會穩定失敗，新的加鎖寫法則乾淨通過。

## [1.7.9] - 2026-05-02

### 修正 — 含星期欄位的排程都差一天觸發（嚴重正確性問題）

- 客戶回報 auto-verify 設為 UI 預設選項「每個月第一個週六 03:00」（cron `0 3 1-7 * 6`），結果 2026-05-02 週六當天並未觸發，下次觸發顯示為 2026-05-03 週日。
- 根因：APScheduler 的 `CronTrigger.from_crontab` 把星期欄位編號為 `0=週一, 6=週日`，但本專案 UI 預設值、README、所有使用者心智模型都依 POSIX cron 慣例 `0=週日, 6=週六`。差一天結果是 `* * * * 6` 跑成週日、`* * * * 0` 跑成週一。受影響的兩個 UI 內建預設值：

  - `0 3 1-7 * 6` 標籤「Monthly (1st Saturday 03:00)」實際在第一個週日觸發。
- 修法：在 `glogarch/scheduler/scheduler.py` 新增 `posix_cron_to_apscheduler()`，把 cron 字串的 dow 欄位先依 POSIX→APScheduler 慣例轉換，再交給 APScheduler。支援單一數字、範圍、列表、萬用字元、步進表達式、星期名稱；轉換後跨週的範圍（例如 POSIX 週五到週一 `5-1` → APS `4-6,0`）會自動在週末邊界拆開。`apply_schedule()`（runtime 註冊）與 `_schedule_to_dict()`（UI 顯示的「下次執行」也是用 cron 算的）統一走這個 helper，所以畫面顯示與實際觸發日從此一致。資料庫中已存的 cron 都依 POSIX 解讀，下次服務啟動就會正確套用。新增 17 條單元測試 `tests/test_posix_cron.py`。
- **對既有排程的影響**：任何有用到星期數字的排程，從今以後會在使用者真正期望的那一天觸發。沒有星期數字的排程（`0 3 * * *`、`0 0 * * *` 等）完全不受影響。先前因為這個 bug 而不知不覺把錯位當作預設行為的客戶，本次更新後實際觸發日會對齊到 cron 字串字面上的意思，沒有資料損失或副作用。

## [1.5.0] - 2026-04-11

### 修正 — OpenSearch `_id` fielddata circuit breaker（嚴重）

排程 OpenSearch 匯出在大 index（650K+ docs）上持續失敗：
`circuit_breaking_exception: [fielddata] Data too large, data for [_id] would be [1.6gb], which is larger than the limit of [1.5gb]`。三個
index（graylog_489、490、492）每晚都失敗。

根因：`search_after` 分頁用 `{"_id": "asc"}` 做 tiebreaker sort。
以 `_id` 排序會強迫 OpenSearch 把整個欄位載入 heap-resident
fielddata — 680K 筆文件 ID 佔了 1.6 GB，超過 circuit breaker 預設
限制。

修法：tiebreaker 從 `_id` 改成 `_doc`（index 寫入順序），零成本，
不需要 fielddata。驗證：graylog_495（680K docs）現在 3m53s 匯出
完成，零錯誤。

### 修正 — OpenSearch 暫態錯誤 retry

`OpenSearchClient._request()` 之前只在連線錯誤（ConnectError /
ConnectTimeout）時重試。HTTP 500、502、503、429 回應直接 raise，
不會重試也不會 failover 到下一台 host。

修法：暫態 HTTP 錯誤會做 exponential backoff retry（最多 3 次，
間隔 1s/2s/4s），耗盡後才 failover 到下一台。非暫態錯誤（4xx）
仍然直接 raise。

### 變更 — 通知格式全面改版

- 內文行拿掉 emoji — 只在標題行放一個狀態 emoji
  （✅ 成功、⚠️ 部分錯誤、❌ 失敗）
- 每個統計值獨立一行，乾淨的 `標籤: 值` 格式
- 錯誤訊息中的長 URL 自動縮為 `<url>`，避免在聊天軟體中斷行
- 匯入通知加上耗時
- 標題範例：`✅ 匯出成功`、`⚠️ 匯出完成（有錯誤）`、`❌ 驗證失敗`

### 修正 — Preflight `collect_field_schema` 無法處理壓縮 schema（code review）

`json.loads()` 直接解析 `field_schema` 欄位，但 `ArchiveDB.record_archive()`
會把大型 schema 壓成 `zlib:…`。導致 preflight 對大型歸檔無聲地 fallback 成
`{}`，mixed-type field conflict 偵測失效。改為使用
`ArchiveDB.decompress_schema()` 解壓，解析失敗時 `log.warning` 而非無聲地
吞掉。backfill 路徑也改用 `_maybe_compress_schema()` 確保儲存一致。

### 修正 — `_dt_to_str()` / `_str_to_dt()` timezone 處理（code review）

`replace(tzinfo=None)` 只拔掉 tzinfo 卻沒先轉 UTC。`+08:00` 的 datetime
會被當成 UTC 存進去，絕對時間偏移 8 小時。改為先
`astimezone(timezone.utc)` 再 strip。內部全用 `datetime.utcnow()`（naive
UTC），既有 DB 資料不受影響。

### 修正 — Cross-conflict 偵測漏掉自動建立的 numeric mapping（code review）

`get_current_custom_mapping()` 之前只讀 Graylog custom field mappings API。
OpenSearch 自動建立的 numeric mapping 看不到，preflight 會漏掉
cross-conflict（target=long，archive=string）。改為先查 active write index
的實際 OpenSearch mapping（`GET /<deflector>/_mapping`），custom mappings
作為補充。

### 新增 — 55 筆單元測試（pytest）

首個公開測試套件。涵蓋：密碼脫敏（10）、DB datetime round-trip（5）、
field_schema 壓縮（6）、DB 重建/備份（5）、清理競態（3）、bulk 匯入
機制（7）、並發匯入鎖（5）、通知格式（7）、OpenSearch `_doc` 排序（1）、
`/api/health` 結構（2）、preflight conflict 計算（4）。

### 修正 — 文件與實作一致性（reviewer 回報）

- **FastAPI `version` 硬寫 `"1.3.1"`**（`web/app.py`）而非讀
  `glogarch.__version__`。改為讀取唯一版本來源。
- **匯出 metadata 的 `glogarch_version` 硬寫 `"1.3.1"`**（`export/exporter.py`
  與 `opensearch/exporter.py`）。改為讀 `__version__`，讓歸檔檔案

- **Config 搜尋路徑 `/etc/glogarch/`** 與安裝指令碼的

  （home 目錄 fallback 也改為 `~/.jt-glogarch/`）。

## [1.4.2] - 2026-04-10

v1.4.0 強化版發行後在 Graylog 7 目標 (.83) 端對端測試時又抓到兩個
潛在的架構 bug 與幾個 UX 問題，本版本一次處理。

### 修正 — Bulk 模式匯入後 Graylog Search 看不到資料

症狀：bulk 匯入回報「已完成，159,286 筆訊息」，OpenSearch 也確實有
資料（`jt_restored_2026_04_09` 索引，166 MB），但在 Graylog 上搜尋
`jt-glogarch Restored (jt_restored)` stream 顯示 0 筆結果。發現 32
萬筆訊息都成了 UI 看不到的殘留索引。

根因：`BulkImporter._index_name_for_doc()` 之前是用每筆文件的
`timestamp` 推算目標索引名稱（`jt_restored_YYYY_MM_DD`）。Graylog
是用 MongoDB 追蹤 index set 的索引清單（`jt_restored_0`、
`jt_restored_1`...），**不是用 `<prefix>_*` wildcard**。Graylog 不認得
不是它自己建的索引，即使名稱開頭對得上也不會列入 stream 搜尋範圍。
Stream → index_set → MongoDB 清單查詢永遠不會回傳那些日期分區索引。

修法：bulk 寫入現在**永遠走 Graylog 管理的 deflector alias**
(`<prefix>_deflector`）。OpenSearch 會把 bulk request 自動路由到 Graylog
標記為 `is_write_index=true` 的底層索引，所以：
- Graylog Search 立刻就看得到我們的文件（寫進 Graylog 自己追蹤的
  寫入索引裡）
- Graylog 自己的 SizeBased / TimeBased rotation 還是繼續適用
- 不會再有殘留索引

`_ensure_index()` 學會偵測 deflector alias 結尾，改用 HEAD 驗證而不是
PUT（否則會 fail with `invalid_index_name_exception`）。舊版預先掃描
全部文件以建立日期索引的邏輯也整個拿掉了。

### 修正 — Bulk 匯入完成「去哪查看」提示被吞掉

症狀：bulk 匯入成功，Graylog stream 也正確建好，但 Web UI 匯入完成
modal 只顯示「已完成！ (N 記錄數）」，沒提示要去 Graylog 哪裡找資料。
後端確實有把 notice 寫進 `jobs.error_message`（直接查 DB 確認過），
但前端就是不顯示。

根因 (`web/routes/api.py::get_job`)：從 Web UI 觸發的 job 會把 SSE
事件累積到 `_job_progress[job_id]`。`/api/jobs/{id}` endpoint 一律
**優先回 in-memory 版本而完全不查 DB**。in-memory 的 `error_message`
是 `last.get("error")` — 成功時是 None，**根本不會去讀 DB 的真正
error_message column**。所以 where_msg 寫了卻永遠回 null。

修法：in-memory 顯示 job 已完成（`phase=done` 或 `pct>=100`）時，改
讀 DB 裡的 row。這樣會回正確的 `error_message`、正確的 `job_type`、
正確的 status。in-memory cache 還是用於正在進行中的 polling。

### 修正 — `/api/jobs/{id}` 把 import 誤回成 export

`_job_progress` 捷徑的另一個副作用：之前硬寫 `"job_type": "export"`
不管實際是什麼 job。從 Web UI 觸發的 import 在這個 endpoint 都被
誤標成 export（列表 endpoint 與 Job History 是用另一條程式路徑所以
通常看不出來）。現在改從 DB row 讀真正的 type。

### 新增 — Verify 排程「立即執行」按鈕

排程頁面之前只有 export 與 cleanup 排程有「立即執行」按鈕，verify
排程被遺漏 — 純粹是 JS render 條件少寫了 verify。後端
`POST /api/schedules/{name}/run` 三種類型本來就支援。`app.js` 一行
修改解決。

### 變更 — Bulk batch_docs 預設 5000 → 10000

v1.4.1 在 .83 (Graylog 7) 測試驗證 — 每個 `_bulk` request 10k 筆跑
得很順，沒有遇到 429 backpressure。對多數 target 可以直接讓吞吐量
翻倍。共改 4 處：`BulkImporter.DEFAULT_BATCH_DOCS`、`web/routes/api.py`
body 預設、`index.html` modal `value`、`cli/main.py --batch-docs` 預設。

### 文件 — Bulk 模式速率 slider 沒有作用

匯入 modal 的「Batch Delay (ms）」slider **只**對 GELF 模式有用。
`BulkImporter.import_archives()` 的 hot loop 沒有任何 inter-batch
sleep — 只有 OpenSearch 回 429 時的 retry backoff。這條 slider
原本就在 `#gelf-mode-fields` 內，選 bulk 模式時整個 div 會 hide，
所以使用者不會在 bulk 模式看到它。Bulk 真正能調的旋鈕是
`batch_docs`。

- **DST 與 APScheduler**:APScheduler 依系統時區運作；像 `0 3 * * *`

### 修正 — Bulk 模式匯入後在 Graylog UI 上看不到資料

## [1.2.0] - 2026-04-09

### 新增 — 合規匯入流程（零遺失保證）

本次重大更新導入「零訊息遺失 + 零 indexer failures」的匯入合規流程。已透過
67 份歸檔、828 萬筆訊息端對端驗證通過。

- **歸檔時記錄欄位 schema** — `archives.field_schema` 欄位（JSON）儲存
  `{欄位名稱: [出現過的型別]}`。OpenSearch 與 API 兩個 exporter 都會在
  `StreamingArchiveWriter` 寫入訊息時順便累積（每筆 ~10 µs，可忽略）。
  匯入 preflight 直接讀 DB，不需要重新掃檔
- **`glogarch/import_/preflight.py`** 新模組，匯入前**先**做完所有防範：
  1. 驗證目標 Graylog API 帳密
  2. **Cluster health check** — 對方 OpenSearch RED 直接 abort
  3. **GELF input 檢查** — 必須存在於指定 port 且為 RUNNING 狀態。同時警告
     `override_source`（會蓋掉原始 source）、`decompress_size_limit`、
     `max_message_size` 等設定問題
  4. **Journal 壓力檢查** — 對方 journal 已有 >10 萬未處理就警告
  5. **Capacity 檢查** — 讀對方 index set 的 rotation/retention strategy，
     估算這次匯入會用到幾份 index；如果**刪除型 retention 會把剛匯入的資料
     刪掉**就直接 abort
  6. **欄位 schema 收集** — 從 DB `field_schema` 欄位讀（毫秒）。1.2.0 之前
     的舊歸檔自動 fallback 解壓掃檔，並 backfill 進 DB
  7. **衝突偵測** — 只 pin 兩種情況的欄位為 keyword：（a）歸檔內同時有
     numeric 與 string 值（內部矛盾，必衝突），或（b）對方目前的 mapping
     是 numeric 而歸檔有字串值。避免過度 pin 撞到 1000 欄位上限
  8. **OpenSearch 欄位上限突破** — 自動 PUT 一個名為 `jt-glogarch-field-limit`
     的 OpenSearch index template，把 `index.mapping.total_fields.limit`
     拉到 10000，徹底解決「Limit of total fields [1000] has been exceeded」
     導致 index rotation 失敗的問題
  9. **套用 custom mappings + cycle** — 透過 Graylog
     `PUT /api/system/indices/mappings`（每個欄位一次 PUT、`rotate: false`），
     套完所有衝突欄位後再 cycle 一次 deflector，新 index 才會吃到新 mappings
  10. **等待新 index 啟用** — poll 直到對方準備好接資料
- **匯入後自動對帳** — GELF 送完後自動撈 Graylog `indexer failures` 數字並
  跟匯入前的 baseline 比對。差異 > 0 就把 compliance violation 訊息寫進
  `jobs.error_message`，讓使用者能立即發現
- **強制目標 Graylog API 帳密** — 匯入對話框必填
  `target_api_url` + (`target_api_token` 或 `target_api_username` + `target_api_password`）。
  前後端都會驗證，缺欄位不執行。同一組帳密用於 preflight、journal 監控、對帳
- **通知欄位敏感資料遮罩** — 所有憑證欄位（Bot Token、Chat ID、Webhook URL、SMTP 密碼、Nextcloud token/帳密等）預設以遮罩顯示，欄位右側有眼睛按鈕可切換顯示/隱藏
- **側邊欄 logo 連結** — 點擊側邊欄左上的 `jt-glogarch` 標題會在新分頁開啟專案 GitHub 頁面
- **歸檔時間軸視覺化**（歸檔清單頁面）
  - 每日分布長條圖，高度依記錄數量比例
  - 拖曳選取時間範圍（小時等級精度），自動填入篩選並套用
  - 滑鼠 hover 時有垂直虛線跟隨，整個欄位區都可選取
  - Hover 提示顯示日期、歸檔數、記錄數、檔案大小
  - 清除選取按鈕（永遠保留位置避免版面跳動）
  - 圖表有邊框，accent 色長條，紅色標記無資料日
- **表格載入動畫** — 篩選或重新載入完成時表格會閃爍 accent 色光暈
- **匯入流量控制系統**
  - Web UI 即時暫停/繼續
  - 即時速率調整（批次延遲滑桿）
  - **三種監控模式** 防止目標 Graylog Buffer/Journal 爆掉：
    - 無（手動控制）

### 修正 — 合規流程驗證時發現的 bug
- **OpenSearch exporter 跨 index 邊界資料遺失** — `is_time_range_covered()` 會把同一次 OpenSearch run 內的姊妹 index 互相擋住，導致跨越 index rotation 邊界的小時內訊息只寫入其中一個 index 的部分。症狀：某些小時的歸檔筆數明顯偏低。修法：加上 `exclude_stream_id_prefix` 參數，讓跨模式 dedup 只擋跨模式不擋同模式內的姊妹。**這個 bug 造成受影響小時 ~17% 資料遺失**。
- **Web UI 匯入對話框狀態殘留** — `_batchImportIds` 在 POST 後立刻被清空（不論成功失敗），導致匯入失敗後重試時靜悄悄什麼都沒做。修法：清空時機延後到 `closeImportModal()`。
- **Web UI 匯入對話框進度殘影** — 失敗匯入的進度條與錯誤文字會留到下次嘗試。修法：在 `watchJob()` 開頭清空 bar/text。
- **Web UI 匯入對話框重試時 host 還是舊值** — 跟上一條相關，新版會在每次重試時重新讀表單值。
- **OpenSearch 欄位上限被撐爆** — 套用大量 custom field mappings 時，Graylog 自動產生的 index template 超過 OpenSearch 預設的 1000 欄位上限，導致 `Limit of total fields [1000] has been exceeded`，graylog index rotation 失敗。Preflight 現在會自動 PUT 一個 override template `jt-glogarch-field-limit` 把上限拉到 10000。
- **目標端殘留舊 custom field mappings** — preflight 中途失敗會在 Graylog MongoDB 留下半套狀態，影響後續 index rotation。README 補上清理流程。
- FastAPI 路由順序：`/archives/timeline` 須在 `/archives/{archive_id}` 之前（422 錯誤）
- pip install 快取問題：`build/` 目錄導致安裝舊版（改為強制重裝）
- 排程編輯 modal：`keep_indices` 值未在 coverage widget 載入後還原
- 時間軸單擊不再殘留舊選區（會把高亮移到點擊位置 1 小時寬的範圍）
